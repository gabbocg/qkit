# Spec A — Rename `qbeamer` to `qkit` and Add CV Format

**Date:** 2026-05-28
**Status:** Revised 2026-05-28 (Option B pivot during Phase 1 implementation)
**Successor work:** Spec B will add a `qkit-manuscript` format under the single `qkit` extension. Out of scope here.

## Revision history

- **2026-05-28 initial:** Proposed two separate extensions `qkit-beamer/` and `qkit-cv/` with YAML keys `qkit-beamer` and `qkit-cv`.
- **2026-05-28 revision:** During Phase 1 implementation, the prescribed YAML keys failed to render — Quarto resolves a YAML `format: <ext>-<fmt>` key by splitting at the LAST hyphen, so `format: qkit-beamer` is parsed as extension `qkit` + base format `beamer`, NOT as extension `qkit-beamer`. Confirmed experimentally: `qkit-beamer` folder + YAML `qkit-beamer-beamer` works; YAML `qkit-beamer` fails with `Unable to read the extension 'qkit'`. **Pivot to Option B**: single extension folder `_extensions/qkit/` contributing both `beamer` and `pdf` formats, with subdirectory organization for the format-specific assets. Clean YAML keys (`qkit-beamer`, `qkit-pdf`) and matches the broader Quarto extension ecosystem convention (titlepages-pdf, letter-pdf, etc.). The CV is invoked as `format: qkit-pdf` — slightly less self-documenting than `qkit-cv` but acceptable; the skeleton's filename (`cv.qmd`) and the project template name (`qkit CV`) carry the semantic naming.

## 1. Motivation

The R package currently named `qbeamer` ships a single Quarto extension that produces a Beamer presentation. The owner wants to broaden the package into a small toolkit of useful Quarto building blocks — starting with the existing presentation and a new CV format ported from a private RMarkdown template. The new identity is `qkit` ("a small set of useful Quarto things").

The user's CV currently lives at `~/Dropbox/Documentos/Research/CV/` as `rmarkdown-cv.Rmd` + a full Pandoc-template `latex-cv.tex`. That source is **not** migrated into the package; only a generic, feature-demonstrating skeleton is bundled.

## 2. Goals

1. Rename the R package `qbeamer` → `qkit` with a hard cut (no compatibility shims).
2. Generalize the install/scaffold API so the user picks a format at install or scaffold time.
3. Migrate the existing Beamer extension as the `beamer` format contributed by the new `qkit` extension, with no visual changes.
4. Add a `pdf` format contributed by the same `qkit` extension that produces a LuaLaTeX CV with the styling features from the user's existing RMarkdown CV.

## 3. Non-goals

- `qkit-manuscript` (Spec B).
- Pkgdown site, GitHub Actions CI, `NEWS.md`.
- Deprecation shims, re-exports, or aliases keeping `qbeamer` callable.
- Visual redesign of the Beamer format.
- Renaming the GitHub repository (the owner handles that on github.com; the spec only updates URLs in `DESCRIPTION`).
- Migrating the user's actual CV content; that stays in the private `Research/CV/` directory.

## 4. User stories

- *As a presenter*, I run `qkit::install_extension()` in a Quarto project and get the `qkit` extension under `_extensions/qkit/`. I write `format: qkit-beamer` in my QMD and render via `qkit::qkit_render("slides.qmd")` — same flow as the old `qbeamer` package.
- *As a CV author*, I run `qkit::create_project("my-cv", type = "cv")` and get a directory containing `index.qmd` (a feature-demonstrating skeleton with `format: qkit-pdf`) plus `_extensions/qkit/`. I render with `qkit::qkit_render("my-cv/index.qmd")` and produce a PDF matching the visual style of the reference CV.
- *As an RStudio user*, `File > New Project` shows two `qkit` entries — one for a Beamer presentation, one for a CV — each with its own wizard.
- *As a kit author*, I can add `"manuscript"` as a third format value in Spec B without changing function signatures.

**Key change from initial spec**: one extension folder `qkit/` contributes both formats. Subdirectories `beamer/` and `cv/` hold the format-specific LaTeX assets (preamble and partials). Paths in `_extension.yml` reference these via relative paths (e.g., `beamer/preamble.tex`).

**RMarkdown templates** (`inst/rmarkdown/templates/`) were dropped from Spec A entirely — the RStudio project templates cover the "new from template" entry point and the duplication was friction. They may be reconsidered later.

### 5.2 R API surface

All four exported functions live under the `qkit_*` / verb-noun naming used today:

```r
install_extension(path = ".", overwrite = FALSE)
```
- No `type` argument — under Option B there is exactly one extension folder (`qkit`) shipping both formats, so a `type` arg would be a no-op.
- Resolves to `system.file("extdata", "_extensions", "qkit", package = "qkit")` and is copied to `<path>/_extensions/qkit/`.
- If the target directory exists and `overwrite = FALSE`, emit a message and skip.
- Returns invisibly the target path.

```r
create_project(path, type = "beamer", title = "Untitled Presentation",
               author = "Your Name", ...)
```
- `type` is a single string, one of `c("beamer", "cv")`. It controls which skeleton is dropped, not which extension is installed.
- Creates `path`, copies the matching skeleton from `inst/rstudio/templates/project/skeleton/` (`beamer.qmd` for `type = "beamer"`, `cv.qmd` for `type = "cv"`) into `path/index.qmd`. The fixed output name `index.qmd` is used regardless of type so RStudio's "open" hook always points at the right file.
- For `type = "beamer"`, substitutes the `title` placeholder in the skeleton's YAML. For `type = "cv"`, substitutes the `author` placeholder. `title` is ignored for CV; `author` is ignored for Beamer.
- Calls `install_extension(path = path)` to drop the (single) extension.
- Returns invisibly the project path.

```r
qkit_render(input, ...)
qkit_preview(input, ...)
```
- Thin wrappers around `quarto::quarto_render` / `quarto::quarto_preview`.
- Both call `ensure_extension(input)` first.

```r
ensure_extension(input)   # internal
```
- Reads the YAML front matter of `input` by scanning for the leading `---`/`---` block and parsing it with `yaml::yaml.load()`. Wrap in `tryCatch`; on parse error, silently no-op and let Quarto report the issue.
- Inspects the `format:` key (handles both scalar string and named map shapes — see `detect_qkit_formats()` below).
- If any `qkit-*` format is referenced and `_extensions/qkit/` is missing next to `input`, calls `install_extension(path = dirname(input))`.
- If no `qkit-*` format is detected, silent no-op.

```r
detect_qkit_formats(input)   # internal
```
- Returns a character vector of base format names (subset of `c("beamer", "pdf")`) referenced via `format: qkit-<name>` keys in the input file's YAML front matter.
- The `format:` key may be a scalar string (`format: qkit-beamer`) or a named map (`format: { qkit-beamer: default }` / `format:\n  qkit-beamer: default`); the function normalizes both shapes to a character vector before matching against the `qkit-` prefix.
- Returns `character(0)` if no qkit format is referenced, the YAML cannot be parsed, or the file has no front matter.

### 5.3 The single `qkit` extension

One extension folder `_extensions/qkit/` contributes two formats. Format-specific LaTeX assets live in subdirectories `beamer/` and `cv/`. The shared `_extension.yml` declares both formats:

```yaml
title: qkit
author: Gabriel Cabrera Guzmán
version: 0.1.0
contributes:
  formats:
    beamer:
      include-in-header:
        - beamer/preamble.tex
      template-partials:
        - beamer/title.tex
      pdf-engine: xelatex
      navigation: horizontal
      theme: default
      colortheme: default
      aspectratio: 43
      keep-tex: true
    pdf:
      include-in-header:
        - cv/preamble.tex
      template-partials:
        - cv/before-body.tex
      pdf-engine: lualatex
      geometry:
        - margin=0.70in
      fontfamily: mathpazo
      fontfamilyoptions:
        - sc
        - osf
      fontsize: 10pt
      linkcolor: blue
      urlcolor: blue
      keep-tex: true
```

Users invoke the slide format via `format: qkit-beamer` and the CV format via `format: qkit-pdf`. Quarto resolves the extension by splitting `qkit-beamer` at the final hyphen into extension `qkit` + base format `beamer`, finds the `qkit` extension, and dispatches to the matching contributed format.

#### 5.3.1 Beamer assets (`_extensions/qkit/beamer/`)

Verbatim relocation of the existing `qbeamer` extension's `preamble.tex` and `title.tex`. No content changes — these are byte-identical to today's files from `_extensions/qbeamer/`, only moved into the new subdirectory.

### 5.4 CV assets (`_extensions/qkit/cv/`)

#### 5.4.1 `preamble.tex`

Ports the "stuff I added" block from the user's existing `latex-cv.tex`, stripped of all Pandoc-template conditionals (Quarto's default template already handles geometry, the base hyperref load, fontenc, inputenc/luatex fontspec). The CV preamble adds CV-specific macros on top. Required contents, enumerated for 1:1 coverage:

**Packages**
- `\usepackage{multicol}` — two-column References block (skeleton body).
- `\usepackage{fontspec}` — explicit load even though Quarto's lualatex template already pulls it; harmless re-load and makes the preamble portable.
- `\usepackage{marvosym}` — `\Letter` and `\Telefon` glyphs used inside the References section body (NOT in the header — the header uses fontawesome icons).
- `\usepackage{fontawesome}` — `\faEnvelopeO`, `\faPhone`, `\faGithub`, `\faGlobe` icons in the title header. Requires TeXLive ≥ 2015.
- `\usepackage{orcidlink}` — `\orcidlink{<id>}` for the optional ORCID field in the title header.
- `\usepackage{enumitem}` — list customization for hanging-indent numbered publications (`labelindent=0pt,labelwidth=…,label=\arabic*.,itemindent=0em,leftmargin=2.75em`).
- `\usepackage{calc}` — provides `\widthof{\ref{last-item}}` used inside the enumitem labelwidth above.
- `\usepackage{array, xcolor}` — needed for the custom column types and the gray vertical-rule helper.
- `\usepackage{sectsty}` — section/subsection font overrides.
- `\usepackage{titlesec}` — section spacing.
- `\usepackage{fancyhdr,lastpage}` — page footer.
- `\usepackage{parskip}` — paragraph spacing in lists.

**Macros and definitions**
- `\definecolor{lightgray}{gray}{0.8}` — used by `\VRule`.
- `\newcolumntype{L}{>{\raggedleft}p{0.14\textwidth}}` — right-aligned date column.
- `\newcolumntype{R}{p{0.8\textwidth}}` — left-aligned description column.
- `\newcommand\VRule{\color{lightgray}\vrule width 0.5pt}` — gray vertical rule used as `\begin{tabular}{L!{\VRule width 1pt}R}` separator.
- `\sectionfont{\rmfamily\mdseries\large\bf}` and `\subsectionfont{\rmfamily\mdseries\normalsize\itshape}`.
- `\titlespacing\section{0pt}{12pt plus 4pt minus 2pt}{4pt plus 2pt minus 2pt}` and same for `\subsection`.
- Itemize redefinition: `\renewenvironment{itemize}{...}{...}` to remove bullets and set `\leftmargin{1.5em}`.
- `\pagestyle{fancy}`, `\renewcommand{\headrulewidth}{0pt}`, `\renewcommand{\footrulewidth}{0pt}`, blank `\lhead/\chead/\rhead/\lfoot`. The footer left/center text uses Pandoc partial variables: `\cfoot{\scriptsize $author$ - $title$}` written directly into `preamble.tex`. Because `include-in-header` is itself a Pandoc partial pass, the `$author$` and `$title$` variables are interpolated from YAML at template-render time, before LaTeX ever runs. `\rfoot{\scriptsize \thepage/{\hypersetup{linkcolor=black}\pageref{LastPage}}}` so the page count isn't colored.
- `\urlstyle{same}` — prevent monospace URLs.
- The `pdftitle` "(Curriculum Vitae)" suffix that the user's existing template appended is dropped (Quarto handles `pdftitle` via the document title; the suffix is cosmetic and adds a Pandoc-template conditional we don't want to reintroduce).

#### 5.4.2 `before-body.tex`

A Pandoc partial that renders the centered name + contact block before the document body. Reads the following YAML metadata:

| Field | Type | Default | Render |
|---|---|---|---|
| `author` | string | required | `\centerline{\huge \bf <author>}` |
| `jobtitle` | string | optional | `\moveleft.5\hoffset\centerline{<jobtitle>}` |
| `address` | string | optional | `\moveleft.5\hoffset\centerline{<address>}` |
| `email` | string | optional | `\faEnvelopeO` + mailto link |
| `phone` | string | optional | `\faPhone` + plain text |
| `github` | string | optional | `\faGithub` + `https://github.com/<github>` link |
| `web` | string | optional | `\faGlobe` + `https://<web>` link |
| `orcid` | string | optional | `\orcidlink{<id>}` + ID (`orcidlink` is loaded by `preamble.tex`) |
| `fontawesome` | bool | `true` | If `false`, fall back to `E-mail:` / `Phone:` / `Github:` / `Web:` labels |
| `updated` | bool | `false` | If `true`, append `\| Updated: \today` to the contact line |
| `keywords` | list of strings | optional | Passed to hyperref `pdfkeywords` — Quarto's default template wires this up; no partial code needed |

Each field is conditional (`$if(field)$ … $endif$`), so omitting any one skips its output cleanly.

Per Q4 decision: **feature demo**. Skeleton ships every LaTeX pattern from the reference CV with placeholder content. Sections included, in order:

### 5.5 RStudio project templates

`inst/rstudio/templates/project/`:
- `qkit-beamer.dcf` — Title: "qkit Beamer Presentation". Binding: `create_project`. Passes `type = "beamer"` via a hidden-by-convention parameter. Exposes a `title` parameter in the wizard. `OpenFiles: index.qmd`.
- `qkit-cv.dcf` — Title: "qkit CV". Binding: `create_project`. Passes `type = "cv"` similarly. Exposes an `author` parameter. `OpenFiles: index.qmd`.

Both dcf files point at the shared `skeleton/` directory; `create_project()` picks the right skeleton file (`beamer.qmd` or `cv.qmd`) based on `type` and writes it out as `index.qmd`.

**RMarkdown templates dropped from Spec A.** The original spec considered shipping `inst/rmarkdown/templates/` entries, but they duplicate the RStudio project templates' content and entry-point UX. Skip them for now; reconsider if there's specific demand.

## 6. Open questions resolved during brainstorm

- **Spec scope**: One spec covers rename + multi-format + Beamer migration + CV addition. Manuscript is Spec B.
- **Hard vs soft rename**: Hard cut.
- **Format packaging**: **Revised 2026-05-28** — one extension folder (`_extensions/qkit/`) contributing both `beamer` and `pdf` formats, with format-specific assets organized into `beamer/` and `cv/` subdirectories. The initial plan of two separate extensions failed because Quarto resolves YAML `format: <ext>-<fmt>` by splitting at the last hyphen, making hyphenated extension names ambiguous. The single-extension layout matches the broader Quarto ecosystem convention (e.g., `titlepages-pdf`, `letter-pdf`).
- **CV skeleton flavor**: Feature demo (all LaTeX patterns shown with placeholder content).
- **API shape**: `install_extension()` takes no `type` arg (single extension); `create_project(type = ...)` still uses `type` to pick which skeleton to drop. No per-format functions.
- **CV engine**: LuaLaTeX (matches existing user setup; Beamer stays on XeLaTeX).
- **Title strategy**: `before-body.tex` partial reading Quarto YAML, *not* a full Pandoc document template.

## 7. Validation gates

### 7.1 Mechanical gates (must all pass)

1. `devtools::document()` regenerates `NAMESPACE` and `man/` with no errors. No `qbeamer_*.Rd` files remain.
2. `devtools::check()` returns 0 errors and 0 warnings. Notes are acceptable.
3. `inst/examples/beamer-example.qmd` renders to a PDF of ≥ 1 page via `qkit_render()` with exit status 0.
4. `inst/examples/cv-example.qmd` renders to a PDF of ≥ 1 page via `qkit_render()` with exit status 0.
5. `grep -r "qbeamer" R/ inst/ DESCRIPTION NAMESPACE CLAUDE.md README.md` returns no matches (the rename is complete).

### 7.2 Manual visual review checklist (CV)

After mechanical gates pass, open the rendered `cv-example.pdf` and confirm:

- [ ] Title header is centered, large bold name, with `jobtitle`, `address`, and fontawesome icon row below.
- [ ] Each section heading has a horizontal rule directly below it (`\noindent\rule{\textwidth}{1pt}` pattern).
- [ ] Education / Research / Teaching / Languages sections render the `L | R` two-column tabular with a thin gray vertical rule between the date and description columns.
- [ ] Publications and Working Papers use numbered lists with hanging indents (`1.`, `2.`, etc., where wrapped text aligns under the description, not under the number).
- [ ] References section is laid out in two columns via `multicols`.
- [ ] Footer shows author + title on one side and `page/total` on the other; the total page count is not colored as a hyperlink.
- [ ] URLs render in upright (not monospace) font.

## 8. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Pandoc default LaTeX template conflicts with custom `preamble.tex` macros | Use `include-in-header` (Quarto loads these *after* Pandoc's preamble), avoid redefining packages Pandoc already loads |
| `fontawesome` LaTeX package requires TeXLive ≥ 2015 | Document this requirement in the format's README/example header; not a code change |
| `mathpazo` font with `sc, osf` options can fail under certain TeXLive distros | Mirrors user's existing working setup; if it breaks for downstream users, they override via YAML |
| User's existing `qbeamer` projects break post-rename | Acceptable per user decision (hard cut, pre-1.0 package) |
| `ensure_extension()` YAML parse fails on malformed front matter | Wrap in `tryCatch`; on parse error, skip auto-install and let Quarto report the error |

## 9. Implementation order (handoff to writing-plans)

This spec deliberately does *not* prescribe step-by-step implementation. The writing-plans skill will sequence the work into a plan with review checkpoints. Suggested phases for the planner to consider:

1. Rename foundation (DESCRIPTION, function renames, extension directory rename, format key updates) — verify Beamer still works.
2. API refactor (`install_extension()` typeless, `create_project(type = ...)` picks skeleton, `ensure_extension()` + `detect_qkit_formats()` YAML scan).
3. CV format assets under `_extensions/qkit/cv/` (preamble, before-body partial) and `_extension.yml` updated to declare both formats.
4. CV skeleton, example, RStudio dcf.
5. Validation (document, check, render both examples).
